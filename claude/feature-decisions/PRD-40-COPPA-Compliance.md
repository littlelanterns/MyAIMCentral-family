### R-13. Implementation shape: the webhook cannot commit client-held rows — a commit RPC does
PRD Screen 5 says the webhook handler "commits the `family_members` rows that were held in memory" — impossible literally (held rows live in client component state). The client, after `stripe.confirmPayment` succeeds, polls a status endpoint; once the verification row exists, the client calls **`commit_consented_members(payload JSONB)`** — a SECURITY DEFINER RPC (auth-gated per R-10) that atomically inserts the held members + their `coppa_consents` rows (consent requires the new member ids, so member-insert and consent-insert belong in one transaction) and returns the ids so the client resumes the existing post-insert pipeline (PIN hash → `ensure_pin_shadow_account` → trigger-provisioned resources). Screen 7 (acknowledgment path) uses the same RPC with the pre-existing verification. If the webhook lags past the polling window, the PRD's "processed but didn't complete" support message shows and the daily reconciliation job closes the loop.

### R-14. Age transitions cover DOB-bearing members only

Members bracketed `under_13` without a DOB never auto-transition — Screen 8 shows a gentle per-child nudge when the static `age` suggests ≥13, and mom edits the bracket (which routes through the supersede path, never deletion).

## 3. The PRD-40 / PRD-31 Stripe boundary (D-PRD40-2 made exact)

One handler, one client module, two owners in sequence. PRD-31 pack ruling 7 already agrees; this section is the contract both builds cite.

**PRD-40 BUILDS (Slice 2 — the platform's first Stripe code):**
1. `supabase/functions/_shared/stripe.ts` — Stripe SDK init from Supabase secrets (`STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`), `constructWebhookEvent()` signature verification, shared error taxonomy. **Generic — zero COPPA specifics.**
2. `stripe-webhook-handler` Edge Function — deployed `verify_jwt=false` (Stripe sends no JWT; auth = signature verification in code) with a **purpose-routed event switch**: `(event.type, event.data.object.metadata.purpose)` → handler. PRD-40 registers `payment_intent.succeeded` / `payment_intent.payment_failed` × `purpose='coppa_verification'`. Unrouted events: log + `200` (ack — never let Stripe retry-spam a purpose we don't handle yet). `config.toml` entry in the same commit (the PRD-42 lesson).
3. `stripe_webhook_events` dedup/log table — `event_id TEXT PK` (Stripe event id), `type`, `purpose`, `received_at`, `processed_at`, `status`, `error`. Router-level idempotency (`ON CONFLICT DO NOTHING` short-circuit) IN ADDITION to the consumer-level unique constraint on `parent_verifications.stripe_payment_intent_id`. PRD-31 inherits both patterns.
4. `create-coppa-verification-intent` Edge Function — authenticated (in-code, mom-only per R-10), rate-limited via `parent_verification_attempts` (5/hr, 20/day), creates the Stripe Customer if none and the $1.00 PaymentIntent with `metadata: {purpose:'coppa_verification', family_id, parent_member_id}`, statement descriptor `MYAIM VERIFY`.
5. Daily reconciliation cron (Convention #246 `util.invoke_edge_function`) — lists recent succeeded intents with `purpose='coppa_verification'` from Stripe, cross-references `parent_verifications`, flags mismatches to admin.
6. Stripe customer id recording: on `parent_verifications.stripe_customer_id` ONLY. PRD-40 does not write `family_subscriptions`.

**PRD-31 EXTENDS (later, its Slice 2):** subscription products/prices/checkout/Customer Portal; registers its five subscription events in the SAME router; owns `family_subscriptions` lifecycle — when creating checkout it first looks up an existing customer via `parent_verifications.stripe_customer_id` (one indexed read) before creating a new Stripe Customer; may activate `verification_method='subscription_payment'` post-beta (admin toggle). Cancellation never revokes COPPA consent; founding status never waives verification (both already in both PRDs).

**Shared invariants:** ONE `stripe-webhook-handler`, ONE `_shared/stripe.ts`, router switches on `metadata.purpose`, event dedup at the router, secrets in Supabase secrets (test mode until the founder flips live keys), idempotency proven by an E2E duplicate-event probe.

**Founder external action (gates Slice 2 proof):** create the Stripe account (or confirm existing), supply TEST-mode `STRIPE_SECRET_KEY` + `STRIPE_WEBHOOK_SECRET` as Supabase secrets. Live keys are a cohort-2-adjacent step, not a build blocker.
