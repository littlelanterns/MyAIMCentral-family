The headline patterns: Personal Growth drift clusters into **three distinct shapes**. First, a cluster of **deferred infrastructure** — PRD-11B Family Celebration, PRD-19 reports/aggregation pipeline, and PRD-19 document upload all have substantial spec surface with no code counterpart in the live database (6 absent tables across PRD-19 alone, plus `family_victory_celebrations` for PRD-11B). Second, a cluster of **dormant column drift** on aggregation-heavy surfaces — `is_shared_with_partner` on `guiding_stars` / `best_intentions`, `share_with_spouse` / `share_with_family` / `is_private_note` / `sort_order` / `document_id` on `archive_context_items`, and `aliases` / `primary_alias` / `use_alias_in_external` / `external_alias` on `archive_member_settings` — columns that the PRDs require but that the live schema either never migrated or partially migrated. Third, a cluster of **stub-registered placeholders** that are honest deferrals (Craft-with-LiLa on PRD-06, Discover-with-LiLa on PRD-07, bar_graph/streak visualizations, Haiku overview card regeneration on PRD-13, LifeLantern aggregation) — these are Deferred-Document per STUB_REGISTRY convention and do not warrant new findings beyond what's already captured. PRD-18 is the batch's strongest coverage: Builds K through N.2 (completed 2026-04-06 through 2026-04-07) wired all four Phase A–D section types per the Enhancement addendum including 23 of the 32 named section types, carry-forward cron per Convention #171, Morning Insight semantic search per Convention #183, teen `talk_to_someone` disposition per Convention #192–197, and rhythm audience derivation per Convention #189.

Three load-bearing surprises worth calling out above the row tables. First, PRD-11B's STUB_REGISTRY entry at line 120 claims "Family Celebration mode | ✅ Wired | Phase 12" — this is false. The `family_victory_celebrations` table does not exist in the live database (confirmed against [claude/live_schema.md](../claude/live_schema.md)), no `FamilyCelebration*` component exists in `src/` (`grep Celebrate Our Family|FamilyCelebration|Family Celebration` across `src/**/*.{tsx,ts}` returns zero matches), and the four prompt modes (highlights, detailed, individual_spotlight, private_parenting_lens) have no code counterpart. Scope 5 did not flip this — Scope 2 now catches it. Second, the PRD-08 family-messaging sub-schema (`family_messages` + `family_message_recipients` tables from §Data Schema L548–577) is architecturally superseded by PRD-15's `messages` + `conversation_spaces` + `conversation_threads` trio — this is a legitimate Intentional-Document per Addendum supersession pattern, but PRD-08 text still specifies the old shape. Third, the PRD-19 `archive_context_items` column additions (`share_with_spouse`, `share_with_family`, `is_private_note`, `sort_order`, `document_id`) are all absent from the live schema. The PRD-19 Cross-PRD Impact Addendum §PRD-13 documents these as required additions but no migration has landed. This is the same column-drift shape as PRD-31's monetization infrastructure: the spec exists, the code doesn't, and the gap is not currently registered in STUB_REGISTRY.
