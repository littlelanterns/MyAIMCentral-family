# Scope 2 — Batch 1 Foundation Evidence

## Worker cover paragraph

This evidence pass examined the five Foundation PRDs — PRD-01 Auth & Family Setup, PRD-02 Permissions & Access Control, PRD-03 Design System & Themes, PRD-04 Shell Routing & Layouts, and PRD-31 Subscription Tier System — against their code reality. Four addenda were read in full at the start of the pass: PRD-31-Cross-PRD-Impact-Addendum.md, PRD-31-Permission-Matrix-Addendum.md, PRD-Audit-Readiness-Addendum.md (global), and PRD-Template-and-Audit-Updates.md (global). Per Scope 2 PLAN §4.1, addendum supersession was applied to every PRD-31-touching row — the Permission-Matrix addendum materially reshapes the `useCanAccess()` contract and introduces `permission_level_profiles`, both now treated as authoritative over base PRD-31 Layer-2 descriptions. The three-column discrepancy tables below contain 34 rows total across the five PRDs (6 for PRD-01, 9 for PRD-02, 6 for PRD-03, 5 for PRD-04, and 8 for PRD-31).

The headline patterns: most Foundation drift clusters into **intentional-deferral shapes** rather than accidental-drift shapes. PRD-31's monetization surface is almost entirely unbuilt by design — all 6 credit/tier-sampling tables are absent from the live database, no Stripe webhook Edge Function exists, and 6 of 7 PRD-31 screens have no code counterpart — but CLAUDE.md Convention #10 explicitly states that `useCanAccess()` returns true for everything during beta, so this is Deferred-Document rather than Missing. Two named columns on PRD-01's `families` table (`tablet_hub_config`, `tablet_hub_timeout`) are legacy carry-forwards superseded by PRD-14D's `family_hub_configs` table; no code reads the legacy columns. The Permission-Matrix addendum's "Mom's Selection Flow" (Light/Balanced/Maximum member onboarding picker) is structurally unwired in PermissionHub even though the backing `permission_level_profiles` table is migrated and seeded (164 rows). One scope-creep signal: PRD-03 specifies ~38 themes and CLAUDE.md Convention #42 claims 38 themes implemented, but `src/lib/theme/tokens.ts` actually defines 46 theme keys — this needs founder adjudication (codify via PRD update, or drop the extras).

Two load-bearing surprises worth calling out above the row tables. First, SCOPE-8a.F1's COPPA prerequisite note identified the Stripe webhook as an absent dependency needed for PRD-40; this evidence pass confirms the webhook handler does not exist anywhere in `supabase/functions/` — the gap is documented twice now and remains the primary downstream blocker for any future paid-tier work. Second, the `useCanAccess()` / `PermissionGate` wrapping requirement stated in CLAUDE.md Conventions #9–10 and PRD-01 §L813 has very low adoption in the current codebase — `useCanAccess()` appears in only 5 source files and `PermissionGate` in only 9. Beta unlock makes this effectively no-op today, but the convention's forward-looking purpose (so tier gating can be flipped on via configuration rather than rebuild) is undercut when most features never wrap. This surfaces as a batch-level consolidation candidate rather than per-PRD finding.
