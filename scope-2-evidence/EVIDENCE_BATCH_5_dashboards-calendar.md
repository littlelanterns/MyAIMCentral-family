# Scope 2 — Batch 5 Dashboards/Calendar Evidence

## Worker cover paragraph

This evidence pass examined the eight Dashboards/Calendar PRDs — PRD-10 Widgets/Trackers/Dashboard Layout, PRD-14 Personal Dashboard, PRD-14B Calendar, PRD-14C Family Overview, PRD-14D Family Hub, PRD-14E Family Hub TV Mode, PRD-25 Guided Dashboard, and PRD-26 Play Dashboard — against their current code reality. Five addenda were read in full at batch start: PRD-14-Cross-PRD-Impact-Addendum, PRD-14B-Cross-PRD-Impact-Addendum, PRD-14D-Cross-PRD-Impact-Addendum, PRD-25-Cross-PRD-Impact-Addendum, PRD-26-Cross-PRD-Impact-Addendum (PRD-10, PRD-14C, PRD-14E have no matching addenda per §3.5 Glob verification). Per PLAN §7 Rule 8, addendum supersession is applied extensively — the PRD-14B recurrence/event-schema surface, the PRD-26 reveal-style vocabulary, and the PRD-14D Hub config storage were all materially reshaped by downstream decisions captured in CLAUDE.md conventions and the Build M completion record. The three-column tables contain 46 rows across the eight PRDs (PRD-10: 7 · PRD-14: 5 · PRD-14B: 11 · PRD-14C: 4 · PRD-14D: 6 · PRD-14E: 2 · PRD-25: 7 · PRD-26: 4).

The headline patterns: most Batch 5 drift is **addendum/convention supersession of base-PRD text**, with the biggest cluster on PRD-14B where Universal Scheduler integration per Convention #117 + the category/week-start/auto-approve column shape per Conventions #110/#114/#115 collectively reshape the calendar_events schema away from the PRD's original description. Build M (closed 2026-04-13) provides the second-largest supersession cluster, rewriting PRD-26's reveal tile architecture (5 reveal styles → 2 per-segment styles; `reveal_tiles` JSONB preferences array → `task_segments.randomizer_reveal_style` per-segment config). Two deliberate deferral clusters stand out: PRD-14B's image-to-event AI intake surface (the `calendar-parse-event` Edge Function does not exist in `supabase/functions/`; Image OCR + LiLa Guided Intake + Send-to-Calendar AI parsing all unbuilt) and PRD-25's LiLa-tool modals (Homework Help + Communication Coach guided modes are seeded in `lila_guided_modes` via migration 00000000000013 but have no invoking UI surface — classic seeded-but-unwired).

Three load-bearing surprises. First, PRD-14's responsive-column `col_span` feature for sections (PRD-14 Decision #14, PRD L148-155, schema L395 in layout JSONB) has zero code hits in `src/` — every section renders full-width, no code path consumes a `col_span` field, and no UI surface lets mom resize a section. This is a genuine post-PRD gap, not a supersession. Second, PRD-14E `/hub/tv` remains a pure `PlannedExpansionCard` stub ([src/App.tsx:66-70,105](../src/App.tsx#L66)) — all seven PRD-14E MVP checklist items are absent from code. This maps 1:1 to the already-closed Foundation finding candidate SCOPE-2.F8-PRD04-HUB-TV-STUB and is cited rather than re-emitted per §7 Rule 7. Third, the PRD-26 spec says revealed items should write `tasks.source = 'randomizer_reveal'` (PRD-26 Decision #7, Schema enum addition), but the live `tasks.source` query surface does not contain that value — randomizer draws persist via the separate `randomizer_draws` table added in Build J, not via a new source-flagged task. This is a cross-reference pattern with PRD-09A/09B Build J architecture and is flagged as PATTERN-FLAG-FOR-SCOPE-3.
