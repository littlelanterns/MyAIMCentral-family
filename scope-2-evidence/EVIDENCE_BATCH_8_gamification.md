# Scope 2 — Batch 8 Gamification Evidence

## Worker cover paragraph

This evidence pass examined the three Gamification PRDs — PRD-24 Gamification Foundation, PRD-24A Overlay Engine & Gamification Visuals, and PRD-24B Gamification Visuals & Interactions — against their code reality. Five addenda were read in full at the start of the pass: PRD-24-Cross-PRD-Impact-Addendum.md, PRD-24A-Cross-PRD-Impact-Addendum.md, PRD-24A-Game-Modes-Addendum.md, PRD-24B-Cross-PRD-Impact-Addendum.md, and PRD-24B-Content-Pipeline-Tool-Decisions.md. Per Scope 2 PLAN §4.1, addendum supersession was applied — in this batch the load-bearing supersessions run in the opposite direction from Batch 1: the base PRDs (24, 24A, 24B) were written March 2026; Build M (2026-04-13, §3 Overrides #1–#9 in the feature decision file) rewrote the architecture wholesale, replacing the overlay 4-stage progression with a Sticker Book mechanic and the 5-theme visual library with a single Woodland Felt theme. CLAUDE.md Conventions #198–222 codify the Build M architecture as canonical. The three-column discrepancy tables below contain 22 rows total across the three PRDs (12 for PRD-24, 5 for PRD-24A, 5 for PRD-24B).

The headline patterns: **PRD-24 Foundation's schema is ~12% built and that's intentional.** Of PRD-24's 8 specified tables, only `gamification_configs` ships. The other 7 (`gamification_events`, `gamification_rewards`, `reward_redemptions`, `treasure_boxes`, `treasure_box_opens`, `gamification_achievements`, `gamification_daily_summaries`) are absent from the live schema with no migration. This is Build M's explicit scope cut — the "baby step" approach (feature decision file §1) deferred the reward economy and append-only ledger to future builds and invented a Sticker Book substrate (gamification_creatures 161 rows, gamification_sticker_pages 26 rows, member_sticker_book_state 18 rows, member_creature_collection 0 rows, member_page_unlocks 18 rows) that PRD-24 does not name. The `roll_creature_for_completion(p_task_completion_id UUID)` RPC is the authoritative pipeline endpoint per Convention #198 and is wired from four hooks (`useCompleteTask`, `useApproveTaskCompletion`, `useApproveCompletion`, `useApproveMasterySubmission`), but it writes to `family_members.gamification_points` directly rather than through a `gamification_events` append-only ledger. **PRD-24A Overlay Engine is ~0% built.** None of PRD-24A's 5 core tables (`dashboard_backgrounds`, `overlay_instances`, `overlay_collectibles`, `recipe_completions`, `background_library`) and none of the 4 Game Modes Addendum tables (`boss_quests`, `bingo_cards`, `evolution_creatures`, `passport_books`) exist in the live schema. Build M Override #1 explicitly replaced the overlay progression with Sticker Book. The 5 launch visual themes (Pets, Apothecary Bottles, Dragons, Pixel Loot, Mythical Creatures) and 7 game modes (Daily Growth, Category Collection, Recipe & Quest, Stamp Passport, Streak Evolution, Boss Battle, Family Bingo) are all absent — the Build M expansion (Convention #209) shipped 4 *creature earning modes* (random_per_task, every_n_completions, segment_complete, complete_the_day) which are a different architectural model operating on `member_sticker_book_state`, not on overlay instances. **PRD-24B Reveal Type Library is ~40% built but mostly dead code.** The 4 CSS/SVG interactive reveals (CardFlipReveal, ThreeDoorsReveal, SpinnerWheelReveal, ScratchOffReveal), the 4 micro-celebrations (PointsPopup, StreakFire, LevelUpBurst, BackgroundCelebration), StarChartAnimation, TreasureBoxIdle, and ReadabilityGradient all exist as React components in `src/components/gamification/` but are wired only into the `GamificationShowcase` demo at `/dev/gamification` (App.tsx:113). None are consumed by a production flow — `RewardRevealModal.tsx:314` explicitly documents `CssRevealRenderer` as a placeholder rendering a generic `<Gift />` icon with a comment that "Phase 4 or later can wire the named components." The 4 video-based reveals (Treasure Chest, Gift Box, Cracking Egg, Slot Machine) have no video assets and no playback path; instead `reveal_animations` (33 rows) + `reward_reveals` (0 rows) + `reward_reveal_attachments` (0 rows) ship a different reveal-library architecture that doesn't match PRD-24B's Screen 1 interface.

Three load-bearing surprises worth calling out above the row tables. First, **Color Reveal architecturally diverged**: Convention #211 declares Color Reveal is a "1:1 task-linked tally counter" (via `member_coloring_reveals.earning_task_id`), explicitly *not* earning-mode-driven. PRD-24B Screen 10 specifies a mom-configured "reveal strategy" (Sequential/Gradual/Random) feeding the reveal from any qualifying achievement source. The code supports only the task-linked model; the strategies are absent. This is a deliberate Build M product simplification, not accidental drift. Second, **the `lists` table has two conflicting reveal-visual columns**: the live schema carries `reveal_type` (migration `00000000000008_tasks_lists.sql:310`) but PRD-24's Cross-PRD addendum specified a `reveal_visual` column. PRD-24B's Cross-PRD addendum called out this naming collision and asked for reconciliation during build-time; the founder's resolution (per Build M notes, pre-existing column) landed as `reveal_type` with the PRD-24 `reveal_visual` name being the stale one. No code anywhere consumes the 8-value reveal type enum; the live `reveal_type` column is a dormant receptacle. Third, **LiLa bingo card and boss quest generation (PRD-24A Cross-PRD Addendum §Impact on PRD-05) is structurally impossible**: no `boss_quests` or `bingo_cards` tables exist to write to, no Edge Function for generation exists in `supabase/functions/`, and no LiLa guided mode was added for either flow. The addendum claim ("LiLa reads family task data → generates balanced 5×5 card...") has zero implementation surface.
