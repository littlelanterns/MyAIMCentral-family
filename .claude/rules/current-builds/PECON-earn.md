## Key architectural decisions this build locks in

- **Config tables are the source of truth; contract payloads are the override**, not the reverse — same ruling TRKG already made for sticker-book creature/page earning modes. This is a repeatable pattern now, worth naming as a platform convention at close-out (the addendum's §12 already proposes the wording).
- **The routine-completion event is the first NEW consumer of `get_member_day_obligations` since FO-COMMAND-CENTER** (Convention #271/#278 Rider-2) — it must never re-derive "what's scheduled today" inline. This is the single hardest rule to violate accidentally (it would be easy to write a quick day-of-week check instead of calling the RPC) and is called out explicitly in the dispatch's hard rules.
- **One payout choke point for routine economics**: `process_routine_step_completion` is the only place per-step points, per-completion points, AND step prizes get paid — mirroring `award_custom_reward_for_completion`'s structural pattern (one completion → one RPC → idempotent, additive, never-throws).
- **No deed at step granularity.** A 10-step routine must not fire 10 `deed_firings` rows — member-level sources (task, intention) ride the connector via standard contracts; routine-step economics are a direct-RPC pattern, exactly like `award_custom_reward_for_completion` already is for task-level custom rewards.
- **Celebration-only, unconditionally**: no deduction as punishment, no retroactive rate changes, no missed-day surface, uncheck never revokes, raising a daily goal mid-day after it fired never un-fires it.
