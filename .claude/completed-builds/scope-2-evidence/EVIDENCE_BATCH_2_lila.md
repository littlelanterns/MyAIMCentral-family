# Scope 2 — Batch 2 LiLa Evidence

## Worker cover paragraph

This evidence pass examined the two LiLa PRDs — PRD-05 LiLa Core AI System and PRD-05C LiLa Optimizer — against their code reality. One addendum was read in full at the start of the pass: `prds/addenda/PRD-05-Planning-Decisions-Addendum.md` (the only addendum matching PRD-05 per PLAN §3.5; PRD-05C has no matching addendum). Per Scope 2 PLAN §4.1, addendum supersession was applied to every registry-touching row — the Planning addendum's Layer-1 context architecture and `lila_guided_modes` registry-expansion model are treated as authoritative over base PRD-05's 15-mode enumeration and 8-step linear context-assembly pipeline. The three-column discrepancy tables below contain 12 rows total across the two PRDs (8 for PRD-05, 4 for PRD-05C).

The headline patterns: most LiLa drift is **architectural-evolution shapes** rather than accidental-code-drift shapes. PRD-05's base registry lists 15 modes; the live `lila_guided_modes` table holds 43 rows because downstream PRDs (PRD-07/12A/12B/16/19/20/23/29/34/28B/37) each seeded their own guided modes via follow-on migrations — which is exactly the pattern PRD-05 §L398 explicitly documents ("individual feature PRDs register their modes"). Similarly, PRD-05's 8-step linear context-assembly pipeline (L430–439) has been superseded by the 3-layer relevance-driven architecture documented in [claude/ai_patterns.md](../claude/ai_patterns.md) L96–166 and anchored in CLAUDE.md Convention P9 — the Edge Function implementation in [supabase/functions/_shared/context-assembler.ts](../supabase/functions/_shared/context-assembler.ts) is the 3-layer version, and that is the truth going forward. PRD-05C's entire monetization-adjacent surface (Optimizer prompt cards, templates, context presets, usage thermometer, Stripe-adjacent credit packs) is structurally unbuilt by design — all four PRD-05C-authored tables are absent from the live database, the families-table column additions (`ai_credit_balance`, `preferred_platform`, `optimizer_mode_default`) never migrated, and the `optimizer` mode_key exists in the registry but the consumer surface is a stubbed "coming soon" button per [PromptPackDetail.tsx:152](../src/features/vault/components/detail/PromptPackDetail.tsx#L152) — which STUB_REGISTRY L52 already flags as misleading UI. The Optimizer gap is the exact same Deferred-Document shape as Foundation's tier-monetization gap (Batch 1 SCOPE-2.F1), and sits downstream of the same beta-readiness default (CLAUDE.md Convention #10).

Three load-bearing surprises worth calling out above the row tables. **First**, the Crisis Override text is duplicated across two files — `src/lib/ai/system-prompts.ts` L14–31 (client-side constant) and `supabase/functions/lila-chat/index.ts` L39–44 (Edge Function inline constant). The two copies drifted slightly during development: the client copy explicitly lists "abuse" and "911" as separate line items while the Edge Function version collapses them into a single resource enumeration. Both are functionally correct and both include the non-negotiable override per CLAUDE.md Convention #7, but the duplication is a maintenance risk and a seed for future drift; SCOPE-8a.F4 already closed the Translator-specific exemption gap and this row is the natural extension. Flagged as an unexpected finding, not a candidate — the override rule is globally enforced and neither copy is wrong today. **Second**, `src/lib/ai/system-prompts.ts` L149–175 `buildFaithContext()` reads `faithPreferences.response_approach` (a TEXT field) — but the live `faith_preferences` schema ([claude/live_schema.md:857–873](../claude/live_schema.md#L857)) and CLAUDE.md Convention #78 both specify individual boolean columns (`prioritize_tradition`, `include_comparative`, `respect_but_dont_assume`, etc.) with NO `response_approach` column. The Edge Function context-assembler loads the correct columns (`faith_tradition`, `denomination`, `special_instructions`), so the drift is confined to the frontend helper — which appears to be dead code relative to the live LiLa conversation path. **Third**, `lila_tool_permissions` is wired end-to-end (written from [MemberAssignmentModal.tsx:106](../src/features/vault/components/MemberAssignmentModal.tsx#L106), read from [useLila.ts:386](../src/hooks/useLila.ts#L386)) but [claude/live_schema.md:219](../claude/live_schema.md#L219) reports 0 rows — meaning no founder family has yet assigned any LiLa tool to a non-mom member. This is adoption-absence, not code-drift; noted but not a candidate.
