# Scope 2 — Batch 4 Tasks/Studio Evidence

## Worker cover paragraph

This evidence pass examined the four Tasks/Studio PRDs — PRD-09A Tasks, Routines & Opportunities (1,417 lines), PRD-09B Lists, Studio & Templates (914 lines), PRD-17 Universal Queue & Routing System (951 lines), and PRD-17B MindSweep (842 lines) — against their code reality. Three addenda were read in full at the start of the pass: PRD-09A-09B-Linked-Steps-Mastery-Advancement-Addendum.md (615 lines), PRD-09A-09B-Studio-Intelligence-Universal-Creation-Hub-Addendum.md (390 lines), and PRD-17B-Cross-PRD-Impact-Addendum.md (182 lines). Per Scope 2 PLAN §4.1, addendum supersession was applied aggressively — the Studio-Intelligence addendum supersedes the base-PRD three-page model with a "create anywhere, display everywhere appropriate" principle per CLAUDE.md convention #149, and the Linked-Steps addendum adds advancement modes + linked routine steps + `practice_log` dual-write that materially reshape sequential and randomizer semantics (conventions #157–167). PRD-17 is itself a reconciliation PRD that explicitly supersedes PRD-09A's `task_queue` schema with authoritative `studio_queue` and requires PRD-09A text to be updated — that text update has not happened. The three-column discrepancy tables below contain 28 rows total across the four PRDs (10 for PRD-09A, 7 for PRD-09B, 5 for PRD-17, 6 for PRD-17B).

The headline patterns: most Tasks/Studio drift clusters into **addendum-supersession shapes** (code ships the later decision; base PRD text is stale) and **deferred-stub shapes** (Studio Intelligence Phase 2, PRD-10-dependent widget/tracker destinations, post-MVP mastery evidence camera upload). Two recurring structural patterns: (a) PRD-09A never got its post-PRD-17 schema rename — all 10 `task_queue` mentions in PRD-09A text should read `studio_queue` per PRD-17 Schema Reconciliation Note, and PRD-09A still lists feature key `tasks_queue` that doesn't exist in the code's feature-key naming (code ships `queue_modal` / `queue_quick_mode` / `routing_strip` / `studio_queue` per PRD-17). (b) The 14-view prioritization framework specified in PRD-09A §Prioritization Views has 7 implemented views in `src/components/tasks/views/`, 5 more surfaced in ViewCarousel but marked `isPlanned: true` and rendered via `PlannedViewStub`, and ABCDE was explicitly removed per founder decision 2026-04-13 per a code comment. PRD-09A text still inconsistently says 13 and 14 views in different paragraphs and does not acknowledge the ABCDE removal.

Three load-bearing surprises worth calling out above the row tables. First, the Studio Intelligence Phase 1 addendum fix for sequential collection creation shipped 2026-04-06 with full SequentialCreatorModal + `useCreateSequentialCollection` wiring and defensive guards in both `createTaskFromData` (throws loudly on `taskType='sequential'`) and `TaskCreationModal` (console.warns and skips), yet `sequential_collections` still shows 0 rows in `claude/live_schema.md:794` (schema regenerated 2026-04-20). This is genuinely possible — the fix has a clean empty state, and nobody in the founder family has opted into a sequential collection yet — but it cross-refs SCOPE-5.F4 which flagged the original breakage. Citing rather than re-emitting per PLAN §7 Rule 7. Second, PRD-17B's cross-PRD addendum requires `mindsweep_settings` records to be auto-created for Mom/Dad/Independent members on family creation and mom/dad emails to auto-populate `mindsweep_allowed_senders`; live_schema shows 18 family_members but 0 rows in both tables. Either the auto-creation path was never wired to `handle_new_user` / `auto_provision_member_resources`, or it fires only lazily when a member first opens MindSweep Settings. Grep confirms no seed call site. Third, PRD-09A §Data Schema L1078 defines `task_type = 'habit'` as a valid enum value, but no code consumes it — there is no HabitView, no habit-specific rendering, no habit creation path. Per CLAUDE.md convention #13 this should be explicitly stubbed-or-deferred; instead it silently sits in the enum. Cross-check with Scope 5 findings (not currently flagged there).

The autopilot labeling defect — MindSweep routing items to victories/guiding_stars/best_intentions/self_knowledge with `source='manual'` instead of `source='mindsweep_auto'` — is closed via SCOPE-8a.F7 and is cited rather than re-emitted per PLAN §7 Rule 7. Cross-references throughout this file follow the same discipline.
