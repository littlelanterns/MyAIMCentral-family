## Worker cover paragraph

This evidence pass examined the eight Compliance-domain PRDs — PRD-27 Caregiver Tools, PRD-28 Tracking/Allowance/Financial, PRD-29 BigPlans, PRD-30 Safety Monitoring, PRD-35 Universal Scheduler, PRD-36 Universal Timer, PRD-37 Family Feeds, and PRD-38 Blog (Cookie Dough & Contingency Plans) — against their code reality. Seven addenda were read in full at the start of the pass: PRD-27-Cross-PRD-Impact-Addendum.md, PRD-28-Cross-PRD-Impact-Addendum.md, PRD-29-Cross-PRD-Impact-Addendum.md, PRD-30-Cross-PRD-Impact-Addendum.md, PRD-35-Cross-PRD-Impact-Addendum.md, PRD-36-Cross-PRD-Impact-Addendum.md, and PRD-37-PRD-28B-Cross-PRD-Impact-Addendum.md (audited PRD-37-half only per scope clarification — PRD-28B is excluded from Scope 2 per SCOPE-5.F4 Finding A). Per Scope 2 PLAN §4.1, PRD-30 structural absence is closed per SCOPE-8a.F3 and this pass audits only independent UI-stub or settings-surface elements that may exist outside the unbuilt core. Addendum supersession was applied to every PRD-35- and PRD-36-consumer row — both infrastructure PRDs materially reshape downstream scheduling and timing contracts across the platform. The three-column discrepancy tables below contain 38 rows total across the eight PRDs (6 for PRD-27, 8 for PRD-28, 5 for PRD-29, 4 for PRD-30, 4 for PRD-35, 5 for PRD-36, 3 for PRD-37, 3 for PRD-38).

The headline patterns: Compliance density is **deliberately uneven**. The infrastructure PRDs (PRD-35 Scheduler, PRD-36 Timer) are Wired per spec with only minor field-naming drift to flag; PRD-28 (Allowance/Homeschool) is mostly Wired with 4 tables, 2 cron jobs, 2 Edge Functions shipped in migrations 100134/100136/100138; but **four PRDs (PRD-27, PRD-29, PRD-30, PRD-37, PRD-38) are entirely unbuilt at the screen level**, with only PlannedExpansionCard stubs registered in `feature_expansion_registry.ts`. These sparse rows are explicit Deferred-Document classifications — they cluster into one consolidation candidate (`PRD27-29-30-37-38-SCREEN-LAYER-UNBUILT`). Two named columns on `access_schedules` (`recurrence_data` → `recurrence_details`, `special_adult_id` → `member_id`) drift from PRD-35's Data Schema but match the remediation migration 00000000000019; this is spec-text vs implementation drift. A minor STUB_REGISTRY staleness: row 516 claims "Subject Tracking section in TaskCreationModal" is Unwired (MVP) but the section is actually wired at [TaskCreationModal.tsx:1947-1966](../src/components/tasks/TaskCreationModal.tsx#L1947) via `countsForHomework` + `homeworkSubjectIds` fields.

Three load-bearing surprises worth calling out above the row tables. First, PRD-35's base-PRD `access_schedules` schema spec (L331–348) specifies `special_adult_id UUID NOT NULL FK → family_members` and `recurrence_data JSONB` — live schema has `member_id` and `recurrence_details` ([claude/live_schema.md:163-174](../claude/live_schema.md#L163)). This is a systemic field-naming drift introduced by the `00000000000004_universal_scheduler.sql` migration at L13–22 — PRD-35 was not literally implemented. Second, the `homeschool_time_review` LiLa guided mode exists in the DB (migration 100138 seeds it) but per STUB_REGISTRY row 515, the conversational mode is NOT wired — only the `homework-estimate` Edge Function is. This is the same "infrastructure-wired-consumer-unwired" pattern Scope 5 flagged throughout this platform. Third, financial data exclusion from LiLa context assembly (Convention #225) is documented at [_shared/context-assembler.ts:13-14](../supabase/functions/_shared/context-assembler.ts#L13) as a comment but the pipeline enforcement is structural-by-absence — the file simply never calls any of the 4 financial tables. This is Wired-by-omission rather than Wired-by-code, which is acceptable but worth naming.
