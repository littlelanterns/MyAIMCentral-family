## Worker cover paragraph

Walked the PRD-16 Meetings integration surface end-to-end for Scope 3+8b. Identified 11 integration seams collapsing to **5 distinct PRD-16-attributed findings** plus contributions to two cross-addendum patterns. The single most load-bearing finding is **PostMeetingReview's notification call uses `notification_type='system'` and `category='tasks'` instead of the addendum-promised `'meeting_completed'` + `'meetings'`** — meeting completion notifications collapse silently into the Tasks notification stream, breaking the addendum's promise that mom/notification preferences can filter by meeting category. Two beta-Y candidates: (1) **Meeting impressions are NOT actually personal** — RLS allows mom to read all meetings + any participant to read meetings they're in (migration 100146 lines 40-42 + 107-110), and `impressions` is a plain TEXT column on the row. There is NO column-level redaction enforcing Convention #232 ("Impressions are personal — only visible to person who ended"). Any participant SELECTing the meeting sees the impressions field. (2) **Dad permission seam is unenforced at app level** — PRD-16 Edge Cases (line 880-882) explicitly requires application-level rejection when dad starts a parent-child/mentor meeting without per-kid `member_permissions` grants. No such check exists; RLS only enforces family scope, so dad can start any kid's mentor meeting through `useCreateMeeting`.

Live schema has the column named `source_type`, not `source`. Migration `100146:298-301` correctly modifies `calendar_events_source_type_check`. Pure documentation drift — code is right, addendum text is wrong. Worth flagging for orchestrator only because it's part of a broader cross-addendum source-naming pattern.

2. **`meeting_schedule_id` on `calendar_events` is the wrong direction.** Addendum L57 + PRD-16 L600 says "calendar event linked via `source_reference_id` to `meeting_schedules.id`." `MeetingSetupWizard.tsx:297` correctly passes `source_type: 'meeting_schedule'` but **does NOT pass `source_reference_id` = the schedule.id** (the schedule isn't created yet at that point). Reverse linkage `meeting_schedules.calendar_event_id` IS populated (`ScheduleEditorModal.tsx:126`). So one direction wired, one direction broken — calendar event row knows it's "meeting_schedule" type but doesn't know WHICH schedule.

3. **`meeting_agenda_items.source` enum gap vs PRD-16 §Flows table.** PRD-16 line 786 says "PRD-15 Requests: A teen can send a 'Request a meeting' to mom." Enum at migration `100146:255` is only `('quick_add', 'notepad_route', 'review_route')`. **No `'request'` value** for items routed from PRD-15 Requests. PRD-16 says "Not a direct meeting creation — mom decides whether to start one" — so the Request becomes a meeting once mom acts, not an agenda item. The seam is still ambiguous; likely intentional but the addendum doesn't clarify.

4. **`useCreateConversation` for meetings doesn't pass `is_safe_harbor`.** `MeetingConversationView.tsx:75-86` creates the LiLa conversation without `is_safe_harbor`. Defaults to false — correct per Convention #243 (meetings should aggregate). Documented for completeness; no finding.

5. **PermissionGate coverage on Meetings is sparse.** Only `MeetingsPage.tsx:350` wraps a section with `PermissionGate featureKey="meetings_shared"`. PRD-16 What's Done line 957 requires `useCanAccess('meetings_*')` on all surfaces. `MeetingConversationView`, `ScheduleEditorModal`, `MeetingPickerOverlay`, `PostMeetingReview`, `MeetingSetupWizard` have no PermissionGate wrap. During beta this returns true everywhere, so it's invisible — but tier rollout will surface gaps.

6. **`meeting_template_sections` has 31,174 rows in production** — vastly more than per-family seed (4 types × ~6 sections = 24 per family × 3 families = 72 expected). Worth investigating: either lazy-seed is firing on every editor open without idempotency, OR the seed includes deeper customization, OR there's a stress-test residue. Outside scope for this evidence pass but flagged for orchestrator.

## Beta Y candidates

**2 SCOPE-8b primary:**

1. **Seam #2: Meeting impressions privacy boundary unenforced.** `meetings.impressions` is a plain TEXT column on a row whose RLS policies (`meetings_mom_read` + `meetings_participant_read`) grant SELECT on the WHOLE row to mom AND any participant of the meeting. Convention #232 ("Impressions are personal — only visible to person who ended") is enforced ONLY by a SQL comment at migration `100146:21`. Couple meeting impressions are readable by the partner. Mentor-meeting impressions are readable by the teen if they are a participant. Single-finding fix: either move impressions to a separate per-ender table, or add a column-level RLS predicate, or deny SELECT of `impressions` to non-`ended_by` requesters via a view. Marquee privacy promise that any beta couple-tester will violate the moment they close their first couple meeting.

2. **Seam #3: Dad permission gate for parent-child / mentor meetings absent.** PRD-16 Edge Cases line 880-882 explicitly promises "If dad tries to start a parent-child meeting with a child he doesn't have meeting permissions for, the system shows a friendly message: 'This meeting type requires permission from [mom]…'" The application-level check is missing. RLS only checks family scope. A dad-without-permission can call `useCreateMeeting({meeting_type: 'mentor', related_member_id: <kid he can't access>, ...})` and successfully insert — then via `meetings_participant_read` he can read the kid's pending agenda items, run the meeting, and read the summary post-completion. Cross-parent privacy boundary that PRD-02 takes seriously elsewhere.

## Top 3 surprises

1. **Notification meeting-completion writes use `category='tasks'` instead of `'meetings'`.** Every meeting Save & Close in the entire app fires a notification with the Tasks category, so mom's notification preferences UI showing a "Meetings" toggle would have nothing to filter. The fix is one line in `completeMeeting.ts:86-87`. Two seams, both off by a string.

2. **`meeting_template_sections` has 31,174 rows in production with 3 families.** That's ~10,000 sections per family. Most likely cause: lazy-seed firing without idempotency on every editor open. Worker could not locate the auto-seed trigger code path. Either the seed lives in a code path I missed (worth orchestrator probe) or there's a lurking duplicate-write bug.

3. **The `meeting` lila_guided_modes seed restricts to `{"mom","dad_adults"}`, excluding `independent_teens`.** PRD-16 grants teens `meetings_basic` + `meetings_shared` and the addendum line 99 + PRD-16 line 557 explicitly allows teens to participate in mentor + family_council meetings. But the `meeting` mode's role gate excludes them — so teens would be blocked from the very LiLa conversation that powers a Live-mode family council meeting they're invited to.
