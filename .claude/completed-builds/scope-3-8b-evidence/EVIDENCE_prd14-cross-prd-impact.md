## Unexpected findings list (seams not covered in addendum)

1. **Widget data sources are largely uninstantiated** — `dashboard_widgets.data_source_type` and `data_source_ids` are PRD-10 schema; only `ListWidget.tsx:21` actually reads `data_source_ids?.[0]`. The rest of the widget renderers ignore it. PRD-14 incoming-flows table (L462-470) describes Active Tasks/Best Intentions/Calendar/Victories widgets reading from their source tables — they do read source tables, but bypass the `data_source_ids` linkage entirely (each widget hardcodes its own query). This is a PRD-10 concern more than PRD-14, but it surfaces here because the addendum's "Verify widget grid rendering accounts for section ordering" implicitly relies on widgets being pluggable via data_source — and they are not.

2. **Default starter widget condition is a count, not a feature gate.** PRD-14 L207-211 specifies starter widgets conditional on member having ≥1 Guiding Star or Best Intention. Code (`Dashboard.tsx:255-296`) does the count check correctly, but uses `dashboardConfig.preferences.starters_deployed` as the once-per-life flag — if mom adds a Guiding Star AFTER first dashboard load (when `starters_deployed=true`), the GS widget never auto-deploys. Spec is silent on whether the "deploy if ≥1" check should be re-evaluated; current code says no.

3. **Section system has Guided keys but no Guided-shell rendering check.** `dashboardSections.ts:14-17` includes `BEST_INTENTIONS`/`NEXT_BEST_THING`/`CELEBRATE` for PRD-25 Guided Dashboard, but `Dashboard.tsx` (this file) is the personal/adult dashboard and doesn't render those keys. They're rendered in `GuidedDashboard.tsx` (different file). The shared section registry creates a footgun if someone calls `getSections` for a personal dashboard whose persisted layout includes those keys (e.g., from a shell-mode change). No explicit role-aware filter.

## Top 3 surprises

1. **`dashboard_widgets` are completely invisible to LiLa context assembly.** PRD-14's AI Integration section L489-495 PROMISES dashboard awareness in LiLa. Schema has `is_included_in_ai` on widgets per Convention #74. Reality: context-assembler.ts has zero references to `dashboard_widgets`/`widget_data_points`. The Heart toggle on a widget changes nothing about LiLa behavior because LiLa has never been told widgets exist.

2. **`col_span` is in the addendum spec, in the PRD body, in the JSONB doc — and entirely absent from the TypeScript type and the renderer.** `dashboardSections.ts:30-35` defines `SectionConfig` with four fields (`key`, `order`, `visible`, `collapsed`) — no `col_span`. The "responsive section width" architecture (PRD-14 L150-155, addendum L28) is documented across multiple specs and never built.

3. **PerspectiveSwitcher's PRD-14D extension is correctly implemented and the PRD-14 addendum is the stale party.** Most "spec vs code" drift goes the other way (code lags spec). Here the addendum (March 10) was superseded by PRD-14D (March 11) by literally one day, and the code follows the newer addendum. The PRD-14 addendum needs an "Updated by PRD-14D" footnote rather than a code change.
