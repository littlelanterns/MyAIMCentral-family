## Unexpected findings list (seams not covered in addendum)

1. **Hub Mode (kiosk lock with localStorage persistence) is a substantial feature absent from PRD-14D and the addendum.** `FamilyHub.tsx:158-208,452-531` implements `hubModeActive` state, localStorage persistence (`myaim-hub-mode`), `verify_hub_pin` RPC integration, an entry/exit flow, and a floating Lock/Unlock button. The PRD-14D L201-205 spec discusses "Long-press enters edit mode" but never names Hub Mode as a kiosk lock — only Hub Settings PIN protection is in PRD scope. PRD-14E TV Mode discusses "hub lock mode" as a TV concept; the implementation here is a tablet kiosk lock, not a TV feature. This is a code addition the PRD doesn't describe — useful, but undocumented.

2. **`families.hub_config` JSONB and `families.tablet_hub_config` JSONB both still exist in live schema** alongside the new dedicated `family_hub_configs` table. The addendum L74-76 explicitly recommends removing or repurposing `hub_config`; that didn't happen. Two unused legacy columns remain on the much-queried `families` table, paying read cost forever for nothing.

3. **`countdowns` table missing the spec'd `scope` enum + `assigned_member_ids` columns.** PRD-14D L549-573 spec's `scope TEXT 'hub_only'|'whole_family'|'specific_members'` and `assigned_member_ids UUID[]`. Live schema (live_schema.md L601-608) shows neither — only `recurring_annually BOOLEAN`. This means **the entire Decision #12 ("Countdowns as assignable widgets") is unbuildable** — no way to push a countdown to specific members' personal dashboards without those columns. Already partially captured by Batch 5 row 6 ("Match confirmed") — but Batch 5 didn't notice the spec/schema delta on assignment scope.

## Top 3 surprises

1. **PIN-per-intention is fully wired in spec, schema, type, and Settings UI — and silently bypassed at the actual tally call site.** This is the cleanest example of "checked-everywhere-except-where-it-matters" drift surfaced in any cross-PRD evidence pass so far. The PRD even predicted the enforcement location ("in the application layer before the insert") and the application layer has the data and skips the check. Recovery is a one-screen client-side change, but the silent-failure mode means mom believes it works today.

2. **The Hub's central widget grid — "bulletin board" — is a hard stub with no implementation pathway started.** The Hub's mom-experience-goal centers on "what mom sticks to the kitchen wall" (Menu, Job Board, Slideshow Frame, Custom Text). The migration creates feature keys (`family_hub` includes "widget grid"), the section is in the default section order, the Settings page lists "Manage Widgets" as a link — and `HubSectionRenderer` returns null. No `dashboard_configs` family_hub-typed record gets provisioned.

3. **PerspectiveSwitcher overgrants to `special_adult` in two ways simultaneously, contradicting both PRD-14D L442 (Special Adult should have NO perspective tab access) AND L451 (Family Overview is mom-only or dad-with-permission).** Code treats `additional_adult` and `special_adult` as identical for tab composition. EVIDENCE_prd14 marked the perspective switcher as "correctly implemented per PRD-14D" — that verdict was right at the additional_adult layer but missed the special_adult overgrant.
