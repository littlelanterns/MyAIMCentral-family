## Unexpected findings list (seams not covered in addendum)

1. **`approveQueueItem`/`bulkApproveQueue` CHECK violation (Seam #6) is silent because mom never sees the .ics import flow in beta usage to date.** Live schema row counts (`calendar_events` = 34) suggest the broken path isn't being triggered yet. The first family who uploads a .ics file will silently lose every event approval to a CHECK error.
2. **`useCreateEvent` does NOT pass `source_type` at all (useCalendarEvents.ts:198–212).** Falls back to schema DEFAULT `'manual'` — even when called from queue-routing or LiLa-guided contexts. Source attribution is lost for non-mom-typed events created via the modal. This is similar to the **PRD-08/PRD-15/PRD-17B `source='manual'` fallback pattern** the prior evidence files flagged.
3. **`option_group_id`/`option_group_title`/`calendar_subtype` columns** (used by SortTab for penciled-in option groups, migration `100096`) are not described in the PRD-14B addendum at all. They arrived from later spec evolution. Calendar Penciled-In Option Groups is a sizable feature with no addendum coverage. Worth flagging for orchestrator.
4. **`approval_outcomes` notification flow per PRD-15 exists for approve/reject (CalendarTab) but NOT for `/sweep` .ics import approvals submitted by mom herself.** Self-service imports skip the notification. Acceptable but undocumented.
5. **MindSweep → Calendar's `mindsweep_detected` source_type value** appears in `src/types/mindsweep.ts:208` and is consumed by CalendarTab as a discriminator. CalendarTab then DOWNCASTS it to `'review_route'` on insert (line 301), losing the MindSweep audit trail. This is a defensive workaround for the missing CHECK constraint value — but it means the audit trail PRD-17B addendum demanded never materializes in `calendar_events`.

## Beta Y candidates

## Top 3 surprises

1. **`calendar_events.source_type` CHECK violation is silent because nobody is using the .ics import flow yet.** When a single founding family or beta tester actually uploads a .ics file and clicks Approve, every approve will throw. The bug has been latent since migration 100146 (which set the CHECK to 7 values that don't include the values code writes). Fix is one-line: add `'ics_import'` and `'mindsweep'` to the CHECK list.
2. **`calendar_color` column is wired alongside the canonical `getMemberColor()` resolver, NOT instead of it.** Calendar surfaces read `m.calendar_color || getMemberColor(m)` everywhere. The PRD-14B addendum's "consistent everywhere" rationale (L33) was undermined by the later Convention #207 canonical-resolver decision. Now there are two color fields where the addendum promised one. CalendarSettingsModal doesn't even expose `calendar_color` for editing — it can only be set via `family_members` profile UI somewhere else.
3. **`calendar_event_create` guided mode + `calendar-parse-event` Edge Function are BOTH unbuilt, but the addendum names them in 3 different sections.** The mode_key was seeded into `lila_guided_modes` via migration `…000013` UPDATE statements (so it shows up in the registry), but no UI invokes it and no Edge Function knows what to do with it. This is the cleanest example of "DB seed without code" — `tests/guided-mode-registry.test.ts` even asserts the row exists, so the test passes while the feature does nothing.
