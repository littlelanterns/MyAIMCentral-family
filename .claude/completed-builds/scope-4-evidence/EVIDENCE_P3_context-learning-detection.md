# Scope 4 — P3 Context Learning Detection Evidence Pass

## Worker summary

**Pattern definition (ai_patterns.md L195):** "Only trigger Haiku when embedding delta exceeds threshold." The applied-correctly shape per PLAN §2.3 is: the context-learning system compares the current conversation's embedding (or mom's current message) against the family's prior-Archives context; only when the delta exceeds a novelty threshold does Haiku get called to synthesize a new context insight (the "Want me to save that?" offer). `specs/AI-Cost-Optimization-Patterns.md:50` restates the intent: "Before calling Haiku to scan for new info, compare mom's message embedding against existing Archives. If max similarity > threshold, the message is about known topics — skip the Haiku scan. Only call Haiku when something genuinely new is detected."

**Site enumeration method:** (1) grep `supabase/functions/` and `src/` for every form of `context_learning`, `context-learn`, `detected.*fact`, `suggestedMember`, `suggestedFolder`, `novelty`, `learning_trigger`, `centroid`, `delta.*threshold`; (2) grep for every `archive_context_items` INSERT site with `source='lila_detected'`; (3) open `lila-chat` and `lila-*` Edge Functions and scan for post-response Haiku scan logic; (4) grep for cron-scheduled Edge Functions that touch `lila_messages`/`lila_conversations`; (5) cross-reference Scope 5 Entry 68 findings and STUB_REGISTRY row 70; (6) confirm `ContextLearningSaveDialog` component has importers.

**Headline result:** The P3 pattern **cannot be applied because the underlying feature is unbuilt**. There is no detection site in any Edge Function. `lila-chat` ends after streaming the Sonnet reply and writing the assistant message — no Haiku scan, no embedding-delta check, no follow-up "I noticed something new" offer is ever produced. `ContextLearningSaveDialog` exists in `src/components/archives/ContextLearningSaveDialog.tsx` but has **zero importers anywhere in `src/`** — the component is a dead React export. `context_learning_dismissals` has 0 rows in live_schema.md, consistent with a flow that has never fired in production. The STUB_REGISTRY row 70 ("Context Learning write-back — ✅ Wired — Phase 13") is inaccurate; it is a STUB_REGISTRY drift, not a P3 cost-pattern violation. **Proposal:** emit as "Deferred — feature unbuilt" rather than an active SCOPE-4.F finding, and surface the STUB_REGISTRY drift as an orchestrator decision for whether to fold it into this packet or escalate to the STUB_REGISTRY-reconciliation track.

## Unbuilt / deferred

The Context Learning Detection feature as a whole is unbuilt. Evidence:

1. **No server-side detector exists.** `lila-chat` ends after streaming the reply; no Haiku post-scan runs. PRD-05C §Optimizer Flow L298-302 step 9 ("CONTEXT LEARNING CHECK (Haiku — low cost)") is a design step with no implementation.
2. **`ContextLearningSaveDialog` has no importer.** The React component exists as a designed surface but is never mounted. It would be triggered by a `detectedFact/suggestedMember/suggestedFolder` payload from the server that is never produced.
3. **`context_learning_dismissals` table has 0 rows** in production (live_schema.md), consistent with a flow that has never fired. The table itself was created correctly in migration `00000000100035:213-230` (RLS, indexes, policies all present) — it is ready to receive writes, but no writer ever runs.
4. **Feature key `archives_context_learning`** is registered in the `feature_key_registry` (migration `00000000000009:486`) and listed in PRD-13 L839 as TBD tier. No consumer in `src/` calls `useCanAccess('archives_context_learning')`, confirming no gated surface exists.
5. **STUB_REGISTRY.md:70 drift.** Registry row reads "Context Learning write-back | PRD-05 | PRD-13 | ✅ Wired | Phase 13". Scope 5 CROSSCUTTING Entry 68 (`STUB_REGISTRY_EVIDENCE_CROSSCUTTING.md:996-1049`) already flagged this row as ambiguous: "The table has 0 rows per live_schema.md. The stub label 'Context Learning write-back' may refer either to the dismissals dialog or to Convention #75's write-back pattern." Convention #75's "checked somewhere, checked everywhere" aggregation write-back IS wired (it rewrites `is_included_in_ai` to source tables like `self_knowledge` — documented in CLAUDE.md §75 and confirmed by Scope 2 PRD-13 Row 1). That is a **different feature** from PRD-13 Screen 7 + PRD-05C §9 Context Learning Detection, which is unbuilt. Scope 2 PRD-13 Row 6 passed the `context_learning_dismissals` table row as "wired per spec" because it verified table presence per live_schema — not because it verified the detection pipeline.
6. **Cross-scope citation — NOT currently captured in AUDIT_REPORT_v1.md.** No SCOPE-2.F, SCOPE-3.F, SCOPE-5.F, or SCOPE-8.F finding explicitly claims "Context Learning Detection is unbuilt." Scope 5 Entry 68 flagged the stub-label ambiguity but did not assign a final verdict; WALKTHROUGH_DECISIONS.md does not contain the phrase "Context Learning" or "context_learning". Scope 2 PRD-13 batch passed all 7 PRD-13 rows as Intentional-Document pass-confirmations (EVIDENCE_BATCH_3_personal-growth.md:101-107). **Therefore the unbuilt-feature gap is not upstream-captured and cannot be cited-and-skipped per PLAN §8 rule 8.** The gap needs a home in either Scope 4 (as P3 Deferred) or a late-opened Scope 2/5 reconciliation finding — orchestrator adjudication needed.

P3 as a cost-optimization pattern has no implementation surface to audit. The pattern cannot be applied or miss-applied because the feature itself is a stub.

## Unexpected findings

1. **`ContextLearningSaveDialog` is a dead React export.** It exists as a component with correct-looking logic (folder resolution, dismissal hash, save + skip paths) but has zero importers. It is not rendered in any shell, drawer, modal, or page. If a future build wires the server-side detector, mounting this component into the LiLa drawer would activate the save flow — the component itself is salvageable, not orphaned-with-bitrot. Classification: **Deferred-Code** (built capability awaiting consumer).
2. **STUB_REGISTRY.md:70 asserts "✅ Wired" for a feature whose end-to-end flow is unbuilt.** This is a STUB_REGISTRY drift — the registry confuses "table exists + migration landed" with "feature detects + writes". Scope 5 Entry 68 already flagged the ambiguity but left the final verdict unrecorded. Orchestrator should decide whether the STUB_REGISTRY row flip (✅ Wired → 🔗 Partially Wired "table + component capability present, server-side detector unbuilt" or ⏳ Unwired (MVP)) belongs in this Scope 4 packet or escalates to a Scope 5 post-closure reconciliation.
3. **The `context_learning_dismissals` RLS policy is primary-parent-only** (migration `00000000100035:228-230` creates policy `cld_manage_primary_parent`). Non-mom family members cannot read or write the table. This matches PRD-13 §Visibility (mom-only Archives administration) but implies that if the feature is ever built, the detector surface must only fire in mom's conversations — or the RLS policy must be loosened to allow additional-adult writes. Latent design decision that has never had to be made because the detector doesn't run. Informational only.
4. **Feature key `archives_context_learning` tier is TBD in the registry** (PRD-13 L839). No tier-gating decision has been made; no `useCanAccess` call exists. When the feature is eventually built, tier assignment is a pending open decision. Not a P3 finding.
5. **PRD-05C §9 "CONTEXT LEARNING CHECK (Haiku — low cost)"** explicitly names Haiku as the classifier model — no mention of an embedding-delta pre-gate in PRD-05C itself. The embedding-delta gate is a Scope 4 / cost-optimization-patterns-spec addition (ai_patterns.md L195 + specs/AI-Cost-Optimization-Patterns.md L50). If the feature is built per PRD-05C literally, it would be Haiku-on-every-mom-message without the delta gate, which would miss P3. **This is a design-spec gap:** PRD-05C does not require the delta gate even though the cost-pattern catalog does. Orchestrator may want to track as a spec-reconciliation follow-up (amend PRD-05C to reference the delta-gate requirement, or accept that P3 is the pattern layer and PRD-05C is the product layer).

Per PLAN §5.1, default is one `SCOPE-4.F{N}` finding per pattern at most. For P3, the proposal is **NOT to emit a SCOPE-4.F finding** because the feature is unbuilt — the pattern has no implementation to audit. Three alternatives surface for orchestrator adjudication:

Record the P3 pass as "Deferred — feature unbuilt (PRD-13 Screen 7 + PRD-05C §9 Context Learning Detection). Scope 4 cannot audit pattern application until the feature builds. **Alternative B — Emit as a structural-gap finding tied to the STUB_REGISTRY drift.**

Fix scope: (1) flip STUB_REGISTRY row 70 from ✅ Wired to ⏳ Unwired (MVP) or 🔗 Partially Wired with note; (2) defer detector build to post-MVP or fold into PRD-05C Phase 2. Downside: duplicates what Scope 5 Entry 68 should have handled.

**Alternative C — Split: defer the unbuilt feature; emit the STUB_REGISTRY drift as a hygiene finding.**

**Worker recommendation: Alternative A.** The STUB_REGISTRY row 70 drift is better routed through the Scope 5 post-closure reconciliation track (Scope 5 Entry 68 is the correct home — it flagged the ambiguity and can be reopened to finalize the verdict) than introduced here. Scope 4's job is pattern verification; when the pattern has nothing to verify, the cleanest output is a Deferred note that flags the feature as "cannot audit yet" rather than importing Scope 5's unfinished business.

**Beta Readiness flag proposal:** N regardless of alternative. PLAN §7 exceptions do not apply: there is no Sonnet-on-child-surface, no context-bulk-load, no cost-ceiling concern from an unbuilt feature. When the feature is eventually built and the embedding-delta gate is wired correctly, cost is bounded by the gate itself; when built incorrectly (Haiku-on-every-message), projected per-family cost is still well under the $5/month exception threshold for typical usage.
